Step 1 — #52 atomic bar mutations.
effect_utils._apply_instant_component: all three restore branches are now
single atomic queryset UPDATEs on the character row:
  - restore_vitality:  vitality_current = Least(F('vitality_current') +
    magnitude, F('vitality_max'))
  - restore_longevity: same shape on the longevity fields
  - restore_acuity:    conditional toward-baseline Case/When capped at
    F('acuity_baseline') both directions, clamped to 0.1-1.9, rounded to
    1 decimal via Cast-to-numeric + Round (Postgres has no
    round(double, int)); the branch then refresh_from_db()s the field so
    the message prints the persisted value.
Refresh-before-display: cmd_use already re-fetches via
get_character_fresh() (which reassigns self.character) after applying the
effect and builds its status payload from the fresh object — the display
side needed no change; the corruption was entirely the stale write.

AUDIT SWEEP — every write to a character bar *_current field outside the
tick engine, and its disposition:
  1. effect_utils restore_vitality        — CONVERTED (atomic).
  2. effect_utils restore_longevity       — CONVERTED (atomic).
  3. effect_utils restore_acuity          — CONVERTED (atomic).
  4. consumers cmd_spend -> combat_utils.recalculate_bars — AUDITED,
     UNCONVERTED: an absolute set (current := recomputed max, the
     level-up-refill family), not a read-modify-write — it reads stats,
     never bar currents, so there is no stale-add to race. Note for the
     design chat: spending a point in END/STR/WIS mid-combat refills both
     bars to their new max by this design; if that is unwanted it is a
     design ruling, not a code race.
  5. views.py character creation (recalculate_bars on the unsaved
     character) — AUDITED, UNCONVERTED: creation-time, no engine
     concurrency exists.
  6. effect_utils.apply_stat_effect — out of mandate (writes stat_*
     fields, not bars); noted that it is a cached-object RMW on stats
     shared with the engine's expiry reversal, a sibling-shaped risk on
     NON-bar fields, left for a future ruling.
The tick engine's own bar writes are untouched. apply_effect_definition
is shared by the consumer and the engine's apply_npc_effects path; the
atomic branches are strictly safe for both callers.

Regression test (tests/test_combat_state.py, HealLostUpdateTests): tick-
engine-style damage persisted while the consumer holds a stale cached
character; heal applied through the consumer path; asserts persisted
result == damaged_value + magnitude (65, not the old code's resurrected
100). Plus clamp-at-max, longevity, and two acuity cases (persisted-value
basis; never overshoots baseline).

Step 2 — #64 one authoritative NPC order.
Core: order_by('spawned_at', 'pk') added to every room/session NPC
queryset — consumers.get_live_npcs_in_room, get_aggro_npcs_in_room,
_fight_enemies, and the tick engine's fight-payload and load_participants

round messages, personal kill lines, focus-shift lines, and fight-panel

Singles render byte-identically (npc_display_name falls back to
npc_display when the name is unique) — the #79/#81 pinned strings hold.
Test (NpcOrderingTests): two identical NPCs — listing order, resolver

Step 3 — #17 respawned NPCs engage immediately.
CONFIRMED MECHANISM: exactly as diagnosed — the tick engine's respawn
path creates the NpcInstance and stops; no engagement check exists
anywhere in the engine, and aggro fires only in the consumer's arrival
paths (movement/flee). No state flag was lagging; the check simply never
ran. Fix implemented at the true cause: process_npc_respawn now ends
with an engagement stage (engage_respawned_aggro) that runs ONLY when
the batched sweep actually created instances — zero recurring per-tick
queries (#107 discipline preserved; the sweep remains four fixed queries
per tick). Contract implemented:
  - Aggressive+attackable respawns into rooms with living, connected,
    non-dying characters engage that tick: the character's active session
    gains the room's unengaged aggro NPCs, or a new session is created
    (first_attacker='npc') — multi-NPC sessions exactly as walk-in aggro
    builds them.
  - Connectedness = the consumer's Redis presence keys (shyland:online:
    <pk>), checked with one mget only when a respawn fired — a respawned
    aggressor cannot engage (and then kill) an offline character.
  - Grammar: the respawn path emits no introduction line, so the
    engagement line introduces — indefinite article via npc_display(...,
    introduction=True): "A brindled testrat snarls and moves to attack!"
  - Ordering mirrors #81: placement, then engagement lines (broadcast to
    the room, dying players excluded via exclude_pks), then fight panel +
    combat-red status last.
Tests (RespawnAggroTests, engine-path via asyncio with the send funnels
and presence check patched): occupied room engages that tick (session,
line text, indefinite article, fight active, in_combat status); empty
room engages nothing; passive respawn engages nothing.
